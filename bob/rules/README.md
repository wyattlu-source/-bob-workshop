# Rules

Rules 定義的是「怎麼做事」的偏好設定——程式碼風格、架構限制、命名慣例這類會**持續**影響 Bob 回應方式的規則。跟 AGENTS.md 不同的地方在於：Rules 可以只針對某個模式生效，成本也比 AGENTS.md 低一些，適合放比較具體、可執行的規範。

## 放在哪裡

```
.bob/rules/              ← 所有模式共用的規則，對話開始時載入
.bob/rules-{mode}/        ← 只在對應模式啟用時載入（例如 rules-plan/、rules-code/）
```

全域規則可以放在使用者層級（跨專案共用），工作區規則優先於全域規則。實際載入順序（由低到高）：

1. 全域規則
2. 工作區「模式專用」規則
3. `AGENTS.md`
4. 工作區一般規則

## 建議寫法

一個規則檔案只講一個主題，短、聚焦，方便之後單獨修改或移除：

```markdown
# TypeScript 規範

## 型別定義
- 優先使用 interface，避免濫用 any
- 善用 generics 表達可重用的型別

## 命名規範
- 介面：PascalCase
- 函數：camelCase
- 常數：SCREAMING_SNAKE_CASE
```

## 最佳實務

- **一個檔案一個主題**：與其寫一份長長的規則文件，不如拆成多個小檔案（例如 `naming.md`、`error-handling.md`），需要調整時只動對應的檔案。
- **模式專用規則放對地方**：只有規劃階段才需要的限制（例如「先列計畫、不要直接改檔案」），放進 `rules-plan/` 而不是共用的 `rules/`，避免在其他模式下也套用不必要的限制。
- **定期審查**：規則會隨對話持續載入、持續影響回應，過時或不再需要的規則要記得移除，不然只是白白增加成本又可能跟新規範互相矛盾。

## 常見錯誤

- 把只在特定情境才成立的規則寫成「所有模式共用」，導致其他情境下 Bob 的行為被不必要地限制住。
- 規則寫得太長、太像一篇教學文章——Rules 是要持續生效的，內容愈長，每次對話的固定成本就愈高。

## 跟 Workshop 的對應

這個 Workshop（Medical）主要練習 Prompt 設計本身，還沒有用到 Rules。如果後續 Workshop 需要示範 Rules 用法（例如統一某產業案例的程式碼風格），對應的 `04-bob-tips.md` 應該連結回這裡，而不是重複說明設定細節。
