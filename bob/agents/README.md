# AGENTS.md

`AGENTS.md` 放在專案根目錄，是四種自訂機制裡**成本最高、範圍最廣**的一種：每次對話開始都會自動載入，而且是整個工作區共用（不分模式）。用來寫團隊層級「為什麼要這樣做」的原則，而不是逐條列出「怎麼做」的細節。

## 什麼時候該用

- 團隊共同遵守、跟任何一個模式都無關的核心原則（例如「先寫測試再改程式碼」「PR 一定要附上驗證步驟」）。
- 新成員（或新 Workshop 學員）第一次打開專案時，最需要先知道的背景與限制。
- 跟安全或合規相關、不能被個別對話忽略的硬性規定。

**不適合放進 AGENTS.md 的內容**：語言風格細節、單一模式才需要的規則、只有特定任務才用得到的操作流程——這些分別交給 Rules、Rules（模式專用）、Skills 處理，避免 AGENTS.md 變成什麼都塞的百科全書。

## 建議寫法

```markdown
# 團隊規範

## 核心原則
1. 品質優先於速度，但不能用「之後再補測試」當理由跳過驗證。
2. 涉及使用者資料的改動，一律先過安全檢查清單再合併。

## 詳細規範
@./.bob/memory/core/coding-standards.md
@./.bob/memory/security/checklist.md
```

重點是**保持簡短**：把會頻繁引用、篇幅較長的規範拆到 `.bob/memory/` 底下的檔案，用 `@path/to/file.md` 語法引用進來，而不是全部寫在 `AGENTS.md` 本文裡。這樣新對話載入時只解析必要的模組，而不是每次都吃下一份完整的百科全書。

## 常見錯誤

- **寫成流水帳**：把每個模式、每種例外都列進去，結果變成一份沒人會完整讀的長文件。應該只留下「不管做什麼都要遵守」的原則，其餘交給 Rules 或 Skills。
- **跟 Rules 分工不清**：AGENTS.md 講「為什麼」，Rules 講「怎麼做」。如果一條規則明顯只是風格偏好（例如命名慣例），放到 Rules 比較合適。
- **忘記引用檔案的路徑要對得上**：`@path` 語法找不到檔案時通常不會中斷對話，但也不會真的載入內容——修改路徑後記得確認引用真的生效。

## 跟 Workshop 的對應

這個 Workshop（Medical）目前不涉及 AGENTS.md 的教學，屬於後續進階 Workshop 的範圍。如果某個產業案例的 `04-bob-tips.md` 需要示範 AGENTS.md 用法，應該連結回這裡，而不是重複說明設定細節。
