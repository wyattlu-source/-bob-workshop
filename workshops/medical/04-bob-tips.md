# Bob 操作技巧

這個 Workshop 主要練習的是 Prompt 設計與程式碼審查，不涉及 AGENTS / Rules / Skills 等進階能力（那些會在後續 Workshop 或 [`bob/`](../../bob/) 的最佳實務文件中介紹）。這裡整理的是這個階段用得上的操作習慣。

## 這個階段會用到

- 逐步、小範圍的 Prompt，而不是一次要求整個系統
- 檢視 Bob 產生的 diff，確認改動範圍跟 Prompt 描述的一致
- 請 Bob「只提出計畫、先不要動手」，確認方向後再放行

## 操作步驟

1. **一次一個 Prompt，一次一個功能**：完成 Prompt 1 並確認畫面正確後，再送出 Prompt 2。不要把五個 Prompt 一次貼給 Bob——範圍太大時，出錯了也很難判斷是哪一步的問題。
2. **送出 Prompt 前，先確認起始檔案是最新狀態**：如果你手動改過 `app.js`，先讓 Bob 讀取目前檔案內容再繼續，避免它根據舊版程式碼做出不一致的修改。
3. **檢查每一次改動的範圍**：Bob 回覆後，先看它改了哪些檔案、哪些函式，確認沒有動到 Prompt 沒有要求的部分（例如 Prompt 2 不應該動到 Prompt 1 寫好的 `render()` 排序邏輯）。如果動到了，直接請它「把 XX 函式改回原本的邏輯，只保留新增的部分」。
4. **卡住時，把錯誤訊息貼給 Bob，而不是重新描述需求**：瀏覽器 Console 出現錯誤時，直接複製錯誤訊息與出錯的程式碼行號給 Bob，比重新解釋一次需求更容易讓它定位問題。
5. **用「先列計畫、再執行」處理重構**（見 Prompt 5）：涉及大範圍調整時，先請 Bob 描述打算怎麼改，你確認後再讓它實際修改檔案，避免一次性的大改動難以審查。

## 常見問題

- **Bob 把已經寫好的功能重寫掉了**：通常是 Prompt 範圍描述得不夠精確。在下一個 Prompt 明確加上「不要修改 XX 函式／XX 邏輯，只新增 YY」。
- **Bob 生成的程式碼跟畫面上的 HTML id 對不上**：把 `index.html` 中相關區塊的原始碼貼給 Bob 一起看，確保它引用的是實際存在的 id，而不是憑推測命名。
- **改了好幾輪之後不確定目前檔案長怎樣**：直接請 Bob「列出目前 app.js 的完整內容」，重新對齊狀態，再繼續下一個 Prompt。
