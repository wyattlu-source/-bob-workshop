# 重點說明

- **Prompt 的顆粒度決定審查的難度**：把「做一個候診看板」拆成五個小 Prompt，每一步都能單獨驗證，是這個 Workshop 最核心的觀念。範圍越大的 Prompt，Bob 產出的程式碼越難逐行確認，出錯後也更難定位是哪個環節造成的。
- **業務規則要寫進 Prompt，不能只講 UI**：Prompt 3「同一時間只能有一位病患看診中」如果只描述成「加一個叫號按鈕」，Bob 很可能不會處理這個例外狀況。把商業需求裡隱含的規則明確寫出來，才能得到符合實際使用情境的程式碼。
- **「只加不改」是保護既有成果的關鍵語句**：在 Prompt 中明確講「不要修改 XX，只新增 YY」，可以大幅降低 Bob 在後續 Prompt 中不小心重寫掉先前已經驗證過的邏輯。
- **重構前先要求列計畫**：Prompt 5 示範了「先問怎麼改、確認後再讓它動手」，這個習慣在改動範圍較大或程式碼已經有人在用時特別重要，能避免一次性的大改動難以審查、難以回退。

## 可延伸應用

這五個 Prompt 的順序（顯示 → 新增 → 狀態轉換與業務規則 → 篩選與統計 → 重構）不是候診看板專屬的，而是大多數 CRUD 型介面的通用開發順序：先讓資料看得見、再讓資料可以被新增、再處理狀態變化與規則、最後才是查詢與彙總。換到其他產業案例（例如銀行的貸款申請看板、零售的訂單管理頁）時，同樣可以照這個順序拆解 Prompt。
