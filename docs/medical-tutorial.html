Bob Workshop 實作教學

智慧診所候診看板 — 跟著做

打開 IBM Bob 的 IDE 環境，依序把下面的 Prompt 貼給 Bob， 從一個空白資料夾開始，一步一步生出完整的候診看板。 每完成一步就重新整理瀏覽器確認畫面。

本頁導覽
開始前準備 Step 1 建立名單 Step 2 掛號表單 Step 3 叫號與完成看診 Step 4 搜尋／篩選／統計 Step 5 重構收尾 驗收 延伸 1 資料持久化 延伸 2 深色模式 進階 AGENTS.md 進階 Rules 進階 Skills 進階 Commands 延伸 3 分科別叫號 課後反思

開始前準備

  1. 建立一個空白資料夾，用 IBM Bob IDE 開啟。
  2. 從 Step 1 開始，依序把 Prompt 複製貼給 Bob，一次一個， 重新整理瀏覽器確認畫面沒問題後再送下一個。

1 建立專案骨架並畫出候診名單

為什麼這樣寫：因為沒有事先準備好的骨架，第一個 Prompt 要讓 Bob 一次把最基本的專案結構、病患資料、畫面骨架生出來。但範圍還是刻意限定在「顯示」， 不要一開始就要求「做出完整系統」，才能先確認 Bob 對病患資料的理解跟你一致，之後每一步都在這個基礎上疊加。
請幫我建立一個全新的靜態網頁專案，只用 HTML + Bootstrap 5（線上引入） + 一般的 JavaScript，不使用任何框架、不需要額外安裝工具。專案結構： index.html assets/css/style.css assets/js/app.js 這是一個診所候診看板。請在 assets/js/app.js 裡建立一份病患資料清單， 先放 3 筆範例病患，每筆包含：姓名、電話、預約時間（HH:MM 格式）、 目前狀態（候診中 / 看診中 / 已完成，先都設成候診中）。 接著把這份資料依照預約時間由小到大排序，顯示成一個表格，每一列 顯示姓名、電話、預約時間、狀態徽章（不同狀態要用不同顏色的徽章 區分），放進 index.html 裡的候診名單表格中。網頁一打開就要自動 顯示這份名單。
預期 Bob 回應：建立 index.html、assets/css/style.css、 assets/js/app.js 三個檔案，並在 app.js 裡寫好病患資料與顯示邏輯。 用瀏覽器打開 index.html，應該能看到 3 筆病患資料依預約時間排序顯示。

2 新增病患掛號表單

為什麼這樣寫：先確認「顯示」正確後，再加「新增」功能。明確告訴 Bob 驗證規則（姓名必填），避免它自己猜測，也避免它順便改動 Step 1 已經確認過的渲染邏輯。
在 index.html 加一個「臨時掛號」按鈕，點擊後跳出一個掛號表單視窗， 欄位：姓名（必填）、電話（選填）、到診時間（預設是現在時間，可以 修改）。送出時： 1. 如果姓名沒填，顯示錯誤訊息，不要送出。 2. 把這位新病患加進候診名單，狀態設為候診中。 3. 關閉視窗、清空表單，讓畫面重新顯示最新的名單。 不要修改我們在上一步已經做好的名單排序與顯示方式。
預期 Bob 回應：新增掛號表單視窗與送出後的處理邏輯，沿用 Step 1 已經做好的排序與顯示方式。 完成後試著送出一筆空白姓名，應該看到錯誤訊息；填姓名後送出，新病患應該出現在名單裡（依時間排序後的正確位置）。

3 叫號與完成看診（含業務規則）

為什麼這樣寫：這裡刻意把「業務規則」寫進 Prompt，而不是只講 UI 行為—— 「同一時間只能有一位病患看診中」是這個功能最容易被忽略的細節，要在 Prompt 裡明確講出來，Bob 才會處理例外狀況。
在候診名單每一列加上操作按鈕： - 狀態是候診中的病患，顯示「叫號」按鈕，點擊後把這位病患改成看診中。 但如果目前已經有別的病患正在看診中，要先跳出一個確認視窗，告知 使用者目前是哪一位病患在看診，使用者確認後，才把原本那位改成 已完成、新的病患改成看診中。 - 狀態是看診中的病患，顯示「完成看診」按鈕，點擊後把狀態改成已完成。 - 狀態是已完成的病患，不顯示任何操作按鈕。 改動後要讓畫面重新顯示最新狀態。
預期 Bob 回應：每一列依狀態顯示對應的按鈕，並處理狀態轉換與例外確認。 測試方式：叫號一位病患後，再對另一位候診中病患按叫號，應該跳出確認視窗。

4 搜尋、篩選與今日統計

為什麼這樣寫：這個 Prompt 涉及多個既有功能（名單顯示、病患資料）的整合， 容易讓 Bob 想要重寫整個檔案。明確要求「只加不改」，降低它動到前面已經驗證過的程式碼的機率。
在候診名單上方加入： 1. 一個搜尋輸入框，輸入姓名關鍵字時，名單要即時只顯示符合的病患 （不分英文大小寫）。 2. 狀態篩選按鈕（全部 / 候診中 / 看診中 / 已完成），點擊後名單只 顯示對應狀態的病患。 3. 四張統計卡片：今日總人數、候診中人數、看診中人數、已完成人數， 數字要隨時反映所有病患目前的實際狀態，不受搜尋或篩選影響。 搜尋與篩選只影響畫面上顯示的內容，不要改動到病患資料本身。 盡量沿用你已經寫好的顯示邏輯，加上篩選條件即可，不要整個重寫。
預期 Bob 回應：名單顯示的地方加入搜尋與篩選判斷，統計卡片獨立計算、不受篩選影響。 測試方式：篩選「已完成」時，統計卡片的「候診中」數字應該維持不變（因為統計不受篩選影響）。

5 重構收尾

為什麼這樣寫：走完前四步後，app.js 可能已經有點亂。 這一步不加新功能，只請 Bob 檢視現有程式碼並提出重構建議，練習「檢查與修改 Bob 的產出」而不是照單全收。
請檢視目前的 assets/js/app.js，看看有沒有重複的內容、命名不一致， 或是可以拆解成更小、更好維護的部分，列出你會怎麼調整，但先不要 直接修改檔案，等我確認後再進行。
預期 Bob 回應：先列出具體的調整建議，等你看過覺得合理後， 再回覆「請照這個計畫修改」放行它實際動手。這一步練習的是「先看計畫、再放行修改」的習慣。

驗收：對照功能檢查清單

走完 Step 1~5 後，你的畫面應該滿足以下所有項目：

  • 開啟 index.html，能看到候診名單表格，依預約時間排序，每列顯示姓名、電話、預約時間、狀態徽章。
  • 點擊「臨時掛號」可以開啟表單，姓名留空無法送出；成功送出後新病患會出現在名單中，狀態為候診中。
  • 候診中的病患可以按「叫號」變成看診中；若已有其他病患看診中，會先跳出確認視窗。
  • 看診中的病患可以按「完成看診」變成已完成，已完成的病患沒有操作按鈕。
  • 搜尋框輸入姓名關鍵字，名單會即時過濾；狀態篩選按鈕可以只顯示特定狀態的病患。
  • 畫面上有四張統計卡片（總人數／候診中／看診中／已完成），數字不受搜尋或篩選影響，永遠反映全部病患的實際狀態。
  • 瀏覽器 Console 沒有錯誤訊息。
卡關對不起來時，把瀏覽器 Console 的錯誤訊息直接貼給 Bob，或請講師協助確認畫面行為是否符合預期。

6 延伸功能：資料持久化 延伸練習

為什麼這樣寫：目前重新整理頁面資料就會恢復成初始範例，這對實際護理站來說不能接受。 要讓 Bob 把資料存在瀏覽器裡，同時要講清楚「清除瀏覽器資料才回到範例」這個邊界條件， 避免 Bob 把預設資料寫死覆蓋掉使用者資料。
目前重新整理頁面後，候診名單會回到最初的範例資料，這不符合實際 使用情境。請幫我加上資料保存功能： 1. 每次病患名單有變動（新增病患、狀態改變）時，把目前的名單存進 瀏覽器的本機儲存空間。 2. 頁面載入時，先檢查本機有沒有存過的資料，如果有就用它取代原本 寫死的範例資料；如果沒有（例如第一次開啟，或使用者清除過瀏覽器 資料），才使用原本的範例資料。 3. 不要改動既有的名單顯示、叫號、完成看診邏輯，只在資料變動時加上 存檔，並在頁面一開始加上讀檔。
預期 Bob 回應：新增存檔／讀檔的邏輯，在名單有變動的地方存檔，並在頁面載入時先嘗試讀檔。 測試方式：新增一位病患後重新整理頁面，該病患應該還在名單中；用瀏覽器開發者工具清除本機儲存資料後重新整理，應該回到最初的三筆範例資料。

7 延伸功能：深色模式 延伸練習

為什麼這樣寫：這個功能牽涉大量既有 CSS，如果只說「加深色模式」， Bob 可能會整個重寫樣式。要求「用 CSS 變數 + class 切換」能限制修改範圍， 並具體點名徽章與統計卡片這種容易被忽略、對比度容易出問題的元件。
在頂部導覽列加一個「深色模式」切換按鈕，讓整個候診看板可以在淺色／ 深色配色之間切換： 1. 切換樣式時盡量重用同一套設計，不要為深色模式另外寫一份完全重複 的樣式。 2. 狀態徽章（候診中／看診中／已完成）與統計卡片在深色模式下要維持 清楚可讀，不能變成背景跟文字顏色太接近、看不清楚。 3. 使用者選擇的模式要記住，重新整理頁面後仍要維持上次選擇的深色或 淺色模式。 不要更動候診名單、搜尋篩選、統計卡片的既有功能邏輯，只處理外觀切換。
預期 Bob 回應：在樣式檔裡定義淺色／深色兩套配色，新增切換按鈕處理深色模式偏好設定的讀寫。 測試方式：切換深色模式後重新整理頁面，應該維持深色狀態；檢查徽章文字在深色背景下是否清楚可讀。

8 進階練習：寫一份 AGENTS.md 進階練習

為什麼這樣寫：走完 Step 1~7，你其實已經跟 Bob 培養出一套默契 （一次只做一件事、業務規則要明講、不要動到已驗證過的邏輯……）。這一步不是新增看板功能， 而是練習把這些「為什麼」寫成一份 AGENTS.md，放在專案根目錄。 之後不管是換一台電腦、換一個人接手，還是隔一段時間再回來改這個專案， Bob 都會自動載入這份原則，不用每次重新溝通一次。
請幫我在專案根目錄建立一份 AGENTS.md，把我們開發這個候診看板專案時 累積下來的原則整理進去： 1. 這是一個 demo／教學用途的專案，病患資料都是假資料，不要加入任何 真實個資，也不要串接真實後端 API。 2. 修改功能時一次只處理一個需求，不要一次重寫整個 app.js。 3. 涉及病患狀態轉換的業務規則（例如同一時間只能有一位病患在看診中） 要明確寫出來，不要讓 Bob 自己猜。 4. 深色模式、資料持久化這類視覺／儲存邏輯的修改，不要影響候診名單、 搜尋篩選、統計卡片既有的功能邏輯。 用條列式寫成簡短的核心原則就好，不要展開成落落長的說明文件。
預期 Bob 回應：在專案根目錄建立 AGENTS.md，條列出上述幾點原則， 內容簡短、每條一句話。測試方式：之後在同一個資料夾開新對話請 Bob 修改功能， 觀察它是否會主動遵守這幾條原則（例如是否還是一次只處理一個需求），而不用你重複交代。

9 進階練習：用 Rules 固定程式碼風格 進階練習

為什麼這樣寫：AGENTS.md 放的是「為什麼」， 像縮排、命名慣例這種持續影響風格、但跟專案原則本身無關的細節， 適合放進 Rules，而不是塞進 AGENTS.md 把它撐大。Rules 也可以之後再慢慢調整， 不用一次寫完美。
請幫我在 .bob/rules/ 底下建立一份 coding-style.md，把我們這個候診看板 專案目前使用的程式碼風格整理成規則，之後每次對話都會自動套用： 1. JavaScript 用 2 個空格縮排，字串一律用雙引號。 2. 每個功能區塊的命名要清楚說明它在做什麼（例如「更新病患狀態」的 部分，命名就要看得出跟更新病患狀態有關）。 3. 每個功能區塊只做一件事，如果內容太長（超過 30 行），建議拆成更 小的區塊。 4. 需要更新畫面時，優先重用既有的顯示邏輯，不要在其他地方另外處理 畫面更新。 寫成簡短的條列規則就好。
預期 Bob 回應：建立 .bob/rules/coding-style.md，列出上述規則。 測試方式：在同一個資料夾開新對話請 Bob 加功能，新產生的程式碼應該自動符合這些風格， 不需要每次都重複交代。

10 進階練習：把「檢查業務規則」包裝成 Skill 進階練習

為什麼這樣寫：Step 3 提醒過「同一時間只能有一位病患看診中」這種 藏在需求背後的業務規則很容易被漏掉。「檢查程式碼有沒有遺漏的業務規則」是一個值得 重複使用的流程，不是一次性的 Prompt——包裝成 Skill 之後，之後任何專案都能重複觸發同一套檢查。
請幫我在 .bob/skills/business-rule-check/ 建立一個 SKILL.md， 定義一個叫 business-rule-check 的技能： --- name: business-rule-check description: 檢查程式碼裡有沒有遺漏的業務規則或例外狀況 --- 執行檢查： 1. 找出程式碼中所有跟「病患狀態改變」有關的地方。 2. 針對每一種狀態改變，列出可能被忽略的例外狀況（例如：如果同時 有兩個人要改成同一個狀態會怎樣？如果資料是空的會怎樣？）。 3. 條列輸出檢查結果，不要直接修改程式碼。
預期 Bob 回應：建立 .bob/skills/business-rule-check/SKILL.md。 測試方式：之後可以請 Bob「用 business-rule-check 這個 Skill 檢查一下 app.js」， 它會依照這個流程列出檢查結果，不用每次重新描述一次要怎麼檢查。

11 進階練習：用 Command 加速常用操作 進階練習

為什麼這樣寫：「請 Bob 列出目前 app.js 完整內容，重新對齊狀態」 是卡關時常用的動作（見 Step 5 之前提到的操作技巧）。這種常常要打、每次描述都差不多的操作， 值得做成一個 Slash Command，一個指令就觸發，不用每次重新打字。
請幫我在 .bob/commands/ 底下建立一個叫 show-state.md 的 Command： --- description: 列出目前 app.js 的完整內容，方便重新對齊狀態 argument-hint: （不需要參數） --- 請列出目前 assets/js/app.js 的完整內容，並簡短說明目前分成哪些 部分、各自負責什麼，讓我可以確認現在的程式碼狀態。
預期 Bob 回應：建立 .bob/commands/show-state.md。 測試方式：之後觸發這個指令（實際觸發方式依 Bob IDE 介面而定）， 就能重複用同一個指令快速看目前檔案狀態，不用每次重新描述需求。

12 延伸功能：分科別叫號看板 延伸練習

為什麼這樣寫：真實醫院有多個科別同時在看診，Step 3 那條 「同一時間只能有一位病患看診中」的規則其實只在單一科別內成立——不同科別應該可以 同時各自有病患在看診。這一步要練習修改既有的業務規則（把範圍從「全院」縮小成 「同一科別」），同時新增一個依科別分組、顯示目前叫號狀況的看板，讓候診區的人 看得懂「現在輪到哪個科別的哪一位」。
請幫我把候診看板擴充成可以區分科別（像真的醫院一樣，掛號時要選科別）： 1. 每一位病患資料加上「科別」這個項目，限定在以下五個科別之一： 內科、外科、小兒科、骨科、皮膚科。 2. 「臨時掛號」表單加一個科別下拉選單（必選，預設「內科」），送出 時一併記錄這位病患的科別。 3. 候診名單表格加一欄顯示科別。 4. 修改叫號的業務規則：原本「同一時間只能有一位病患看診中」，改成 「同一個科別同一時間只能有一位病患看診中」——不同科別可以同時 各自有病患在看診，只有同一科別內才需要跳出確認視窗、把原本看診 中的病患標記為完成看診。 5. 在統計卡片下方新增一個「各科別現況」區塊，列出五個科別，每個 科別顯示目前看診中的病患姓名（沒有病患看診中就顯示「暫無看診」）。 不要更動搜尋、篩選、統計卡片既有的功能邏輯，只在這些地方新增科別相關的顯示與規則。
預期 Bob 回應：每一位病患資料加上科別、掛號表單新增科別選單、 表格新增科別欄，叫號的判斷邏輯改成依科別比對現有看診中病患，並新增一個依科別 分組顯示現況的區塊。 測試方式：分別在「內科」跟「外科」各叫一位病患看診，兩者應該可以同時顯示 「看診中」、不會互相跳出確認視窗；但同一科別內再叫另一位病患，才會跳出確認視窗。

課後反思

走完整個練習後，花幾分鐘想想這幾個問題：

  1. 整個練習中，哪一個 Step 花最多時間跟 Bob 來回調整？是 Prompt 描述得不夠精確，還是需求本身比較複雜？
  2. 如果重新做一次，你會怎麼修改 Step 1 的 Prompt？
  3. Step 3 要求把「同一時間只能有一位病患看診中」這條業務規則寫進去—— 你的專案裡，有哪些類似「藏在需求背後」的規則，是你之前寫 Prompt 時可能會漏掉的？
  4. 這個工作流程（拆解需求 → 小範圍 Prompt → 檢查結果 → 重構收尾）中， 哪些部分你覺得可以套用到自己現在的專案？
