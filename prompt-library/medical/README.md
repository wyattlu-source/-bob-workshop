# Prompt Library — 智慧診所 / AI Clinic

整理自 [`workshops/medical/03-prompts.md`](../../workshops/medical/03-prompts.md)，這裡的版本拿掉了候診看板的細節，改寫成可以套用到其他「清單 + 狀態管理」型介面的通用寫法（例如訂單管理、任務看板、報修單追蹤）。

## 資料渲染 Prompt

**適用情境**：已經有一個資料陣列，要把它渲染成表格或卡片列表。

```
這是一個{{系統名稱}}的專案（{{技術棧}}）。assets/js/app.js 裡的
{{陣列變數名}} 是{{資料主體}}，每筆包含 {{欄位清單}}。

請完成 render() 函式：把 {{陣列變數名}} 依 {{排序欄位}} 排序後，
渲染成 {{目標容器選擇器}} 的列表，每列顯示 {{顯示欄位}}，
{{狀態欄位}} 用不同顏色的徽章呈現。不要更動資料模型本身。
```

**預期輸出**：一個只做渲染、不做其他事的函式，方便單獨驗證。

**可調整參數**：陣列變數名、欄位清單、排序欄位、目標容器選擇器。

## 新增表單 Prompt

**適用情境**：要在既有清單上加一個「新增一筆資料」的表單，且渲染邏輯已經確認過、不希望被改動。

```
在 {{頁面}} 加一個「{{按鈕文字}}」按鈕，點擊後開啟 Modal，
表單欄位：{{必填欄位}}（必填）、{{選填欄位}}（選填）。
送出時：
1. 若{{必填欄位}}為空，顯示錯誤訊息，不送出。
2. 建立新物件（{{預設狀態}}），加入 {{陣列變數名}} 陣列。
3. 關閉 Modal、清空表單、重新呼叫 render()。

不要修改 render() 已經完成的排序與顯示邏輯。
```

**預期輸出**：新增功能只新增程式碼，不動到既有渲染邏輯。

**可調整參數**：按鈕文字、必填/選填欄位、預設狀態。

## 狀態轉換 + 業務規則 Prompt

**適用情境**：清單項目有多個狀態，狀態轉換之間有隱含的業務規則（例如「同一時間只能有一筆處於某狀態」）。

```
在{{清單}}每一列加上操作按鈕：
- 狀態為 {{狀態A}} 的列顯示「{{動作A}}」按鈕，點擊後把狀態改成 {{狀態B}}。
  但如果已經有其他項目是 {{狀態B}}，要先跳出確認視窗，
  說明目前是誰／哪一筆處於 {{狀態B}}，確認後才進行狀態轉換。
- 狀態為 {{狀態B}} 的列顯示「{{動作B}}」按鈕，點擊後把狀態改成 {{狀態C}}。

改動後呼叫 render() 更新畫面。
```

**預期輸出**：狀態轉換邏輯連同業務規則的例外處理一起實作。

**可調整參數**：狀態名稱與轉換順序、業務規則的具體條件。

## 搜尋 / 篩選 / 統計 Prompt

**適用情境**：清單已經可以顯示與新增，要加上查詢能力與彙總數字，且不想讓既有邏輯被整個重寫。

```
在{{清單}}上方加入：
1. 一個搜尋輸入框，輸入時即時用{{搜尋欄位}}過濾名單。
2. 狀態篩選按鈕群，點擊後只顯示對應狀態。
3. 統計卡片：{{統計項目清單}}，數字要即時反映{{陣列變數名}}的目前狀態，
   不受搜尋/篩選影響。

搜尋與篩選只影響顯示內容，不要修改{{陣列變數名}}本身。
盡量重用 render() 現有的邏輯，加上篩選條件即可，不要整個重寫。
```

**預期輸出**：篩選邏輯與統計邏輯分開成兩個函式，避免混在一起。

**可調整參數**：搜尋欄位、統計項目清單。

## 重構收尾 Prompt

**適用情境**：走完前面幾個 Prompt 後，程式碼可能已經有點亂，想在不新增功能的前提下請 Bob 檢視並提出重構建議。

```
請檢視目前的 {{檔案路徑}}，指出有沒有重複的邏輯、命名不一致、
或是可以拆成更小函式的地方，列出你會怎麼重構，但先不要直接改動檔案，
等我確認後再進行。
```

**預期輸出**：一份具體的重構建議清單，等使用者確認範圍後才實際修改檔案。

**可調整參數**：檔案路徑。
